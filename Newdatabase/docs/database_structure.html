คู่มืออธิบายทุกตาราง ทุกคอลัมน์ พร้อมชนิดข้อมูล — สร้างจาก prisma/schema.prisma ฉบับปัจจุบัน (6 ตาราง)

🔭 ภาพรวม — ตารางเชื่อมกันยังไง

หัวใจคือ stock_transactions (สมุดบัญชีเคลื่อนไหว) ตารางอื่นเป็น "บริบท" ที่ชี้เข้าหามัน — เส้น → คือ Foreign Key (ชี้ไปหาตารางอื่น)

item_groups (กลุ่มสินค้า 23) └─ items (สินค้า 2,382) └─ stock_transactions (เคลื่อนไหว 579) ◄── หัวใจของระบบ │ ├─ document_id ──► stock_documents (หัวใบเอกสาร — เริ่มว่าง) │ ├─ created_by / requested_by / resolved_by ──► users │ └─ 1 ใบ ─ n ─► stock_request_items (รายการที่ขอเบิก — เริ่มว่าง) │ └─ item_id ──► items └─ created_by ─────────────────────────► users (คนคลัง + ผู้ขอเบิก มีคอลัมน์ role — เริ่มว่าง)

📖 ชนิดข้อมูล & สัญลักษณ์ (อ่านก่อน)

String = ข้อความ (SQLite เก็บเป็น TEXT)
Int = จำนวนเต็ม เช่น 1, 42
Float = เลขทศนิยม เช่น 10.5
Boolean = จริง/เท็จ (true/false)
DateTime = วันที่ + เวลา
? ต่อท้ายชนิด = ว่างได้ (NULL)
PK Primary Key — บัตรประชาชนของแถว ห้ามซ้ำ
FK Foreign Key — ชี้ไปหาแถวในตารางอื่น
UNIQUE ห้ามมีค่าซ้ำในตาราง
ต้องมี ห้ามเว้นว่าง
ว่างได้ เว้นว่าง (NULL) ได้
อัตโนมัติ database ใส่ให้เอง ไม่ต้องกรอก

คอลัมน์ชนิดคีย์/เงื่อนไขหน้าที่

6. สมุดบัญชีเคลื่อนไหวสต็อก stock_transactions 579 แถว · หัวใจของระบบ

ทุกการรับเข้า/เบิกออก/ยอดยกมา/ปรับยอด = 1 แถวที่นี่ — ยอดคงเหลือ = SUM(qty_change) ต่อสินค้า ไม่มีคอลัมน์เก็บยอดที่อื่น

คอลัมน์ชนิดคีย์/เงื่อนไขหน้าที่
idIntPK เลขรันอัตโนมัติเลขลำดับแถว database ออกให้เอง
item_idStringFK → itemsเป็นความเคลื่อนไหวของสินค้าตัวไหน
typeStringต้องมีOPENING (ยอดยกมา) · IN (รับเข้า) · OUT (เบิกออก) · ADJUST (ปรับยอด) — แอปคุมค่าเอง
qty_changeFloatต้องมีจำนวนที่เปลี่ยน — บวก = เข้า, ลบ = ออก (ทศนิยมได้ เช่น นับเป็นเมตร) → ยอดคงเหลือรวมจากคอลัมน์นี้
unit_costFloat?ว่างได้ต้นทุน/หน่วยของรายการนั้น — NULL = ไม่รู้
projectString?ว่างได้เบิกไปใช้โปรเจกต์อะไร
noteString?ว่างได้หมายเหตุ — สินค้า 64 ตัวที่ยอดติดลบมีคำเตือนอยู่ตรงนี้
transaction_dateDateTimeต้องมีวันที่ของเกิดจริงในโลก (คนกรอก/copy จากหัวใบ) — คงไว้ที่บรรทัดเสมอ เพื่อให้ query ยอดไม่ต้อง join
created_atDateTimeอัตโนมัติเวลาบันทึกเข้าระบบ — database ลงให้เอง (now()) ต่างจากบนได้ เช่น เบิกศุกร์ คีย์จันทร์
document_idInt?FK → stock_documents ว่างได้NULL = ไม่ได้เกิดจากใบ (OPENING 579 แถวเดิม + ADJUST) — IN/OUT ต้องมีใบเสมอ (แอปคุม)
created_byInt?FK → users ว่างได้NULL = ไม่รู้คนทำ (ข้อมูลยกมา) — แอปประทับจากคนที่ login เอง ห้ามให้ผู้ใช้เลือกชื่อเอง

🧠 แนวคิดสำคัญที่ร้อยทั้ง 6 ตาราง

ยอดคงเหลือเป็นค่าคำนวณ — ไม่มีคอลัมน์เก็บยอด

ยอดคงเหลือของสินค้า = SUM(qty_change) ของ item นั้น คำนวณสด ๆ ทุกครั้ง ถ้ามีทั้งคอลัมน์เก็บยอดและประวัติ วันหนึ่งสองที่จะขัดกัน → หลัก "ความจริงอยู่ที่เดียว" (single source of truth)

เบิกออก (ISSUE) ต้องผ่านการอนุมัติ — กดขอ ≠ ตัดสต็อก

ไม่ว่าจะขอเบิกล่วงหน้าผ่านเว็บ หรือเดินมาขอที่ห้อง ทุกคำขอสร้างใบ ISSUE ที่ status=PENDING + พักรายการไว้ที่ stock_request_items — ยังไม่สร้างแถวใน stock_transactions (ไม่งั้นยอดลดทั้งที่ของยังไม่ออก) พอคนคลัง (role=STAFF) กด "ยืนยัน" ถึงสร้าง transaction จริงจาก qty_confirmed แล้วเปลี่ยนเป็น CONFIRMED · ถ้ากดยกเลิก → CANCELLED ไม่มี transaction เลย · ฝั่ง RECEIVE คนคลังทำเองตรง ๆ เริ่มที่ CONFIRMED ไม่ผ่าน PENDING

ยังไม่มีระบบ "กันยอดจอง" — คนคลังเป็นด่านจับ

คำขอ PENDING ไม่สร้าง transaction จึงล่องหนจากยอดคงเหลือและจากคำขอใบอื่น → 2 คนจองของชิ้นสุดท้ายพร้อมกันได้ ระบบไม่เตือน ชนกันตอนคนคลังยืนยันคนที่ 2 (แล้วปรับ qty_confirmed หรือยกเลิก) — เวอร์ชันแรกตั้งใจให้คนเป็นด่านจับ ถ้าอยากได้ "ยอดพร้อมจ่าย = คงเหลือ − ยอดจองค้าง" ค่อยเพิ่ม logic ทีหลัง (ไม่ต้องแก้ schema)

transaction_date ≠ created_at

transaction_date = วันที่ของขยับจริง (คนเลือก, แก้ได้, เอาไปคิดยอด/รายงาน) · created_at = เวลาที่ระบบรับแถว (เครื่องลงให้, แตะไม่ได้, เป็นหลักฐานสืบย้อน) — เหมือนวันไปเที่ยว vs วันโพสต์ลง Facebook

NULL แปลว่า "ไม่รู้/ไม่มี" ไม่ใช่ 0 หรือค่าว่าง

latest_cost = NULL คือ "ไม่รู้ราคา" (ต้องแสดงตามนั้น ห้ามโชว์ 0 บาท) · document_id = NULL คือ "ไม่ได้เกิดจากใบ" · created_by = NULL คือ "ไม่รู้คนทำ" — NULL มีความหมายเสมอ

Soft delete — ห้ามลบแถวจริง

สินค้า/คน ที่เลิกใช้ → ตั้ง is_active = false แล้วซ่อน ไม่ลบออกจากตาราง เพราะประวัติ transaction ยังชี้มาหา และรหัสสินค้าห้ามนำกลับมาใช้ (ป้าย QR จริงติดของอยู่)

onDelete: Restrict — database กันลบใบ/คนที่มีประวัติชี้อยู่

FK ที่ชี้หา users/ใบ (created_by, requested_by, resolved_by, document_id) ตั้ง onDelete: Restrict — ถ้าพยายามลบใบหรือคนที่เคยมีรายการชี้อยู่ database จะ error ทันที เป็นด่านป้องกัน audit trail ขาด (ต่างจากกฎ type/status ที่ SQLite บังคับไม่ได้ — อันนี้บังคับได้จริง)

SQLite ไม่มี enum — แอปต้องคุมค่าเอง

คอลัมน์อย่าง type (OPENING/IN/OUT/ADJUST), doc_type (RECEIVE/ISSUE), status (PENDING/CONFIRMED/CANCELLED) และ role (STAFF/REQUESTER) เก็บเป็นข้อความธรรมดา database ไม่ช่วยกันค่าผิด โค้ดฝั่งแอปต้องตรวจเองว่าใส่ได้เฉพาะค่าที่กำหนด
