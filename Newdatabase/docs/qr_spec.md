# QR Code Spec — รหัส 9 หลักบนป้ายสินค้า

> **ใครอ่าน อ่านเมื่อไหร่:** ผู้พัฒนา repo web ที่จะทำฟีเจอร์สแกน/พิมพ์ป้าย QR — อ่านก่อนเขียน parser และใครก็ตามที่สงสัยว่าเลข 9 หลักบนป้ายของจริงในคลังคืออะไร

เอกสารนี้คือ*สเปคใช้งาน*สำหรับแอปใหม่ — ถ้าอยากเข้าใจระบบ Excel เดิมแบบเจาะลึก อ่าน `warehouse_2568_logic_explanation.md` ใน repo ต้นทาง (โดยเฉพาะข้อ 4, 5, 9)

---

## 1. QR ในโลกจริง — ทำไมเรื่องนี้สำคัญ

ระบบเดิมพิมพ์ป้าย QR ตอน*รับของเข้าคลัง*แล้วติดลงบนของจริง — **ป้ายเหล่านั้นยังติดอยู่บนของในคลังตอนนี้** และจะอยู่ต่อไปอีกนาน

นี่คือเหตุผลของกฎที่แข็งที่สุดในระบบ: **ห้าม reuse `item_id` เด็ดขาด ตลอดกาล** — ถ้าเอารหัส `02001` ของสายไฟที่เลิกใช้ไปตั้งให้สินค้าใหม่ ป้ายบนของจริงในคลังจะสแกนแล้วชี้ผิดตัวทันที การ "เลิกใช้" สินค้าทำได้ทางเดียวคือ soft delete (`is_active = false` — ดู [`data_dictionary.md`](data_dictionary.md) ข้อ 4)

## 2. รูปแบบรหัส 9 หลัก

```
รหัส 9 หลัก (text เสมอ) = MMYY + Item_ID
                          ─┬─┬─   ──┬──
                           │ │      └─ 5 หลักท้าย = item_id (คีย์จริง)
                           │ └─ ปี พ.ศ. 2 หลักท้าย
                           └─ เดือน 2 หลัก
```

ตัวอย่างจริงจากข้อมูล:

| QR | แตกได้เป็น | ความหมาย |
|---|---|---|
| `116615041` | `11` + `66` + `15041` | เดือน 11 ปี พ.ศ. 2566 (= ค.ศ. 2023), สินค้า `15041` |
| `086710118` | `08` + `67` + `10118` | เดือน 08 ปี พ.ศ. 2567 (= ค.ศ. 2024), สินค้า `10118` |

**ปี พ.ศ. (Buddhist Era) คือ ค.ศ. + 543** — `YY` 2 หลักในรหัสคือ 2 หลักท้ายของปี พ.ศ. ช่วง 25YY ดังนั้นสูตรแปลง: **ค.ศ. = 2500 + YY − 543** (เช่น `66` → 2566 − 543 = 2023)

---

## 3. กฎทอง: 5 ตัวท้ายเท่านั้นที่เป็นคีย์

**รหัสเต็ม 9 หลักไม่ใช่ตัวระบุที่ unique และไม่ใช่คีย์ของอะไรทั้งนั้น** — ส่วนที่เชื่อถือได้มีแค่ 5 หลักท้าย (`item_id`) ส่วน `MMYY` ข้างหน้าเป็นข้อมูลประกอบเท่านั้น

ข้อเท็จจริงที่ต้องรู้ (ทั้งหมดเกิดจริงในข้อมูลเดิม):

1. **รหัสเต็มซ้ำกันได้ถูกต้อง** — ของชนิดเดียวกันรับเข้าเดือนเดียวกันหลายครั้ง ป้ายออกมาเลขเดียวกันหมด ไม่ใช่ error
2. **MMYY เป็นเดือน/ปีของ*เหตุการณ์ที่สร้างรหัสครั้งนั้น* ไม่ใช่ของล็อตเสมอไป** — ตัวอย่างจริงจากระบบเดิม (logic explanation ข้อ 9): ของรับเข้าด้วยรหัส `066802001` (มิ.ย. 68) แต่ตอนเบิกออกถูกคีย์เป็น `076802001` (ก.ค. 68) — สินค้าตัวเดียวกัน รหัสเต็มคนละตัว ห้ามเขียนโค้ดที่ assume ว่า MMYY ตรงกับล็อตรับเข้า
3. **ระบบเดิมเคยพยายามใช้รหัสเต็มเป็นคีย์แล้วเจ็บมาแล้ว** — ชีต Output ใช้รหัสเต็ม lookup กลับไปหา Input เพื่อดึงข้อมูลล็อต (วันที่รับ, ต้นทุน) มี 14 รายการที่หาไม่เจอ ข้อมูลล็อตหายไปเลย แต่*ยอดสต็อกไม่พัง*เพราะยอดคำนวณจาก `item_id` 5 ตัวท้ายเท่านั้น (บทเรียนนี้ถูกออกแบบเข้า database ใหม่แล้ว: `stock_transactions` ผูกกับ `item_id` ตรง ๆ ไม่มีคอลัมน์ QR)

## 4. บริบทระบบเดิม (สั้น ๆ)

ระบบ Excel เดิมใช้รหัสนี้ 2 ระดับ: (1) แตก 5 ตัวท้ายเป็น `Item_ID` เพื่อ lookup ข้อมูลสินค้าและคำนวณยอด — ส่วนนี้เชื่อถือได้และย้ายมาเป็นแกนของ database ใหม่ (2) ใช้รหัสเต็ม lookup ข้ามชีตเพื่อดึงข้อมูลระดับล็อต — ส่วนนี้เปราะ (ดูข้อ 3.3) และ**ไม่ได้ย้ายมา** รายละเอียดเต็มอยู่ `warehouse_2568_logic_explanation.md` ใน repo ต้นทาง ข้อ 6–7

## 5. สเปคสำหรับแอปในอนาคต

### 5.1 การ parse (ตอนสแกนป้ายเดิม)

```
input:  ข้อความจากเครื่องสแกน 1 ค่า
กติกา:
1. ตรวจรูปแบบ: ยาว 9 ตัว และเป็นตัวเลข 0-9 ล้วน — ไม่ผ่าน = แจ้ง "รหัสไม่ถูกรูปแบบ"
2. item_id = 5 ตัวท้าย (คง string เสมอ)
3. เอา item_id ไปหาใน items — ไม่เจอ = แจ้ง "ไม่พบสินค้ารหัสนี้" แล้วหยุด
   ห้ามสร้างสินค้าใหม่อัตโนมัติจากการสแกนเด็ดขาด
4. MMYY (4 ตัวหน้า) = ข้อมูลประกอบ ใช้แสดง/บันทึกเพิ่มได้ แต่ห้ามใช้ตัดสินใจเชิงข้อมูล
```

ตัวอย่าง JavaScript:

```js
function parseQr(raw) {
  const qr = String(raw).trim();
  if (!/^\d{9}$/.test(qr)) return { ok: false, error: "รหัสไม่ถูกรูปแบบ (ต้องเป็นตัวเลข 9 หลัก)" };
  const mm = qr.slice(0, 2);            // "01"-"12" — นอกช่วงนี้ถือว่ารูปแบบผิด
  if (Number(mm) < 1 || Number(mm) > 12) return { ok: false, error: "เดือนในรหัสไม่ถูกต้อง" };
  const beYear = 2500 + Number(qr.slice(2, 4)); // พ.ศ. เช่น 2566
  return {
    ok: true,
    item_id: qr.slice(4),               // "15041" — string เสมอ ห้าม Number()
    month: mm,
    year_be: beYear,
    year_ce: beYear - 543,
  };
}
```

### 5.2 ข้อห้ามเด็ดขาด

| ข้อห้าม | เหตุผล |
|---|---|
| ห้ามแปลงรหัส (ทั้ง 9 หลักและ `item_id`) เป็นตัวเลข | เลข 0 นำหน้าหาย: `02001` → `2001` = คนละสินค้า/รหัสพัง |
| ห้ามใช้รหัสเต็ม 9 หลักเป็น unique key หรือคีย์ lookup | ซ้ำกันได้ถูกต้อง + MMYY ไม่นิ่ง (ข้อ 3) |
| ห้าม reuse `item_id` | ป้ายจริงติดของอยู่ในคลัง (ข้อ 1) |
| ห้ามสร้างสินค้าใหม่อัตโนมัติเมื่อสแกนไม่เจอ | รหัสที่พิมพ์ผิด/สแกนเพี้ยนจะกลายเป็นขยะใน master ทันที |

### 5.3 ป้ายของใหม่ — ประเด็นที่ยังไม่ตัดสิน

รูปแบบป้ายสำหรับ*ของที่รับเข้าใหม่*หลังแอปขึ้นระบบ เป็นการตัดสินใจของ repo web (ยังไม่ล็อก) — คำแนะนำจากฝั่ง migration: **คงรูปแบบ `MMYY + item_id` เดิม** เพื่อให้ป้ายเก่า-ใหม่สแกนด้วย parser ตัวเดียวกันได้ และคนหน้างานไม่ต้องเรียนรู้ใหม่ แต่ถ้าจะเปลี่ยน (เช่น ใส่เลขล็อตจริงจัง) ให้ออกแบบให้ parser แยกแยะรูปแบบเก่า/ใหม่ได้จากความยาวหรือ prefix

## 6. ตาราง edge cases สำหรับคนเขียนเทสต์

| กรณี | อินพุตตัวอย่าง | พฤติกรรมที่ถูก |
|---|---|---|
| รหัสปกติ | `116615041` | ok, item_id `15041`, เดือน 11 ปี 2566 |
| เลข 0 นำหน้า item_id | `066802001` | ok, item_id `02001` (string — ห้ามกลายเป็น `2001`) |
| สแกนแล้วไม่เจอสินค้า | `1166` + `99999` | error "ไม่พบสินค้า" — ห้าม auto-create |
| รหัสเต็มซ้ำกับที่เคยสแกน | `116615041` สองครั้ง | ปกติ — ไม่ใช่ error (ซ้ำได้ถูกต้อง) |
| ยาวไม่ครบ/เกิน | `11661504`, `1166150411` | error รูปแบบ |
| มีตัวอักษรปน | `11661504A` | error รูปแบบ |
| เดือนเพี้ยน | `136615041` (เดือน 13) | error รูปแบบ |
| ช่องว่างหัวท้ายจากเครื่องสแกน | `" 116615041 "` | trim ก่อนแล้ว parse ตามปกติ |
