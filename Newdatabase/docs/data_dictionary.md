> **ใครอ่าน อ่านเมื่อไหร่:** ผู้พัฒนา repo web (และคนดูแลระบบในอนาคต) — อ่านก่อนเขียน query หรือฟีเจอร์ใด ๆ ที่แตะ database ไฟล์นี้คือเอกสารอ้างอิงหลักว่าทุกตารางทุกคอลัมน์หมายถึงอะไร และมีกติกาอะไรที่โค้ดต้องเคารพ

**ปรัชญาการออกแบบ: database นี้เป็น "ledger" (สมุดบัญชีเดินรายการ)** — เราไม่เก็บ "ยอดคงเหลือ" ที่ไหนเลย เก็บแต่*เหตุการณ์* (รับเข้า +5, เบิกออก −3, ...) แล้วคำนวณยอดจากผลรวมของเหตุการณ์เอาเสมอ (ดูกฎเหล็กข้อ 8)

ทำไมทำแบบนี้: ถ้าเก็บทั้ง "รายการเคลื่อนไหว" และ "ยอดคงเหลือ" แยกกัน สองที่นี้จะมีวันไม่ตรงกันแน่นอน (บั๊ก, แก้มือ, insert พลาด) แล้วจะไม่รู้ว่าอันไหนถูก — เก็บที่เดียวคำนวณเอาไม่มีปัญหานี้ และได้ประวัติตรวจสอบย้อนหลัง (audit trail) ฟรี นี่คือการตัดสินใจที่ล็อกแล้วข้อ 7 ใน [`CLAUDE.md`](../CLAUDE.md)

**`stock_documents`, `stock_request_items` และ `users` เป็นตารางเปล่าที่เตรียมไว้ให้แอป** — ต่างจาก 3 ตารางข้างบนที่มีข้อมูลจริงจาก Excel สามตารางนี้ไม่มีอะไรให้ import (Excel เดิมไม่เคยมีแนวคิด "ใบเอกสาร", "คำขอเบิก" หรือ "ผู้ใช้ระบบ") จึงเริ่มว่างเปล่า รอแอปทยอยเติมตอนใช้งานจริง — รายละเอียดดูข้อ 6–7 และ **flow ขอเบิก–อนุมัติ ดูข้อ 6.1**

## 2. กติกาการอ่านเอกสารนี้ + รูปแบบการเก็บข้อมูลใน SQLite

ตารางคอลัมน์ด้านล่างใช้หัวตารางชุดเดียวกัน: ชื่อคอลัมน์ / ชนิดใน Prisma / ชนิดจริงใน SQLite / NULL ได้? / ค่า default / ความหมาย+ตัวอย่าง

สิ่งที่ SQLite เก็บจริง (สำคัญมากเวลาเขียน query ตรง ๆ ไม่ผ่าน Prisma):

| ชนิดใน Prisma | เก็บจริงใน SQLite | ตัวอย่างค่าจริงใน db |
|---|---|---|
| `String` | TEXT | `'02001'` |
| `Boolean` | INTEGER `0`/`1` | `1` = true, `0` = false |
| `Float` | REAL | `-1.0`, `3016.85` |
| `DateTime` | **TEXT รูปแบบ ISO-8601** | `'2026-07-03T11:34:48.104+00:00'` (นี่คือตัวอย่าง*รูปแบบ* — ค่าจริงเปลี่ยนตามเวลาที่ insert) |
| `Int` | INTEGER | `1` |

ข้อควรรู้ 2 เรื่องที่พลาดกันบ่อย:

- **รหัสทุกตัวเป็น TEXT เพราะเลขศูนย์นำหน้ามีความหมาย** — `"02001"` ≠ `2001` ห้าม cast เป็นตัวเลขเด็ดขาด ไม่งั้นรหัสพัง (เหตุผลเชิงกายภาพอยู่ใน [`qr_spec.md`](qr_spec.md) ข้อ 1)
- **NULL ≠ 0** — NULL แปลว่า "ไม่รู้ค่า" ส่วน 0 แปลว่า "รู้ว่าเป็นศูนย์" สองอย่างนี้คนละเรื่องกัน เช่น `latest_cost = NULL` คือไม่รู้ราคา (ห้ามแสดงเป็น 0 บาท) — ใช้กับ `latest_cost` และ `unit_cost`

| `updated_at` | `DateTime` | TEXT (ISO) | ไม่ | ⚠️ ไม่มี! | เวลาแก้แถวล่าสุด — **อ่านกับดักด้านล่าง** |

**Soft delete คืออะไร ทำไมห้ามลบจริง:** การ "ลบ" สินค้าในระบบนี้คือ set `is_active = false` ไม่ใช่ `DELETE` แถวทิ้ง — เพราะ (1) รายการเคลื่อนไหวเก่าของสินค้านั้นยังต้องชี้หาชื่อ/ข้อมูลสินค้าได้ (2) database ก็บังคับอยู่แล้ว: FK ตั้งเป็น `ON DELETE RESTRICT` = ถ้ามี transaction ชี้อยู่ SQLite จะไม่ยอมให้ลบแถวสินค้า (ส่วน `ON UPDATE CASCADE` = ถ้าแก้ `item_id` ตัวแม่ ค่าที่ชี้อยู่จะเปลี่ยนตาม — แต่อย่าทำ เพราะห้าม reuse/เปลี่ยนรหัสอยู่แล้ว)

**⚠️ กับดัก `updated_at` สำหรับคนเขียน raw SQL:** คอลัมน์นี้ `NOT NULL` แต่**ไม่มี default ใน database** — คนเติมค่าคือ *Prisma client* (จาก `@updatedAt` ใน schema) ไม่ใช่ trigger ของ SQLite ดังนั้น:
- `INSERT` ตรง ๆ โดยไม่ใส่ `updated_at` → พังทันที (NOT NULL constraint)
- `UPDATE` ตรง ๆ โดยไม่ set `updated_at` → ไม่พัง แต่ค่าจะค้างเก่าเงียบ ๆ
- เขียนผ่าน Prisma → ไม่ต้องคิดเรื่องนี้เลย มันจัดการให้
- เกร็ดเดียวกัน: `created_at` มี `DEFAULT CURRENT_TIMESTAMP` ใน db ก็จริง แต่ format ที่ default ผลิต (`2026-07-03 11:34:48` — เว้นวรรค ไม่มี timezone) *ต่างจาก* format ที่ Prisma เขียน (ISO มี `T` และ `+00:00`) — ถ้า insert ดิบให้ใส่วันที่เองเป็น ISO ให้ตรงกับของเดิม อย่าพึ่ง default

| `qty_change` | `Float` | REAL | ไม่ | — | จำนวนที่เปลี่ยน: **บวก = ของเข้า, ลบ = ของออก** (ทศนิยมได้ เช่น สายไฟนับเป็นเมตร) |

| `document_id` | `Int?` (FK) | INTEGER | ได้ | NULL | ชี้ไป `stock_documents.id` — **NULL มีความหมายเฉพาะ ไม่ใช่ลืมกรอก**: แปลว่า "รายการนี้ไม่ได้เกิดจากใบเอกสาร" (ปัจจุบัน = ทุกแถว เพราะทั้ง 579 แถวเป็น `OPENING`) กติกาที่แอปต้องคุมเอง (database บังคับแบบมีเงื่อนไขไม่ได้): `IN`/`OUT` ต้องมีใบเสมอ, `OPENING`/`ADJUST` ต้องเป็น NULL เสมอ — ดูข้อ 6 |

**`type` เป็น "enum จำลอง" — แอปต้องคุมค่าเอง:** enum คือชนิดข้อมูลที่จำกัดค่าให้เลือกได้แค่ชุดที่กำหนด (เช่น PostgreSQL มี `CREATE TYPE ... AS ENUM`) แต่ **SQLite ไม่มี enum** คอลัมน์นี้จึงเป็น TEXT ธรรมดาที่ database ยอมรับค่าอะไรก็ได้ — โค้ดฝั่งแอปต้องเป็นด่านกันเอง ให้เขียนได้แค่ 4 ค่านี้:

| ค่า | ความหมาย | เครื่องหมาย `qty_change` |
|---|---|---|
| `OPENING` | ยอดยกมาจากการย้ายระบบ (มีอยู่แล้ว 579 แถว — สร้างโดย migration เท่านั้น แอปไม่ควรสร้างเพิ่ม) | + หรือ − |
| `IN` | รับของเข้าคลัง | + เท่านั้น |
| `OUT` | เบิกของออก | − เท่านั้น |
| `ADJUST` | ปรับยอดหลังตรวจนับจริง (stock audit) — ใช้แก้ยอดที่ไม่ตรงกับของจริง รวมถึงเคลียร์ 64 ตัวติดลบ | + หรือ − |

## 6. ตาราง `stock_documents` — หัวใบรับเข้า/เบิกออก

**แบบแผน header–detail:** 1 ใบ (`stock_documents` 1 แถว) มีรายการได้หลายบรรทัด (`stock_transactions` หลายแถวที่มี `document_id` เดียวกัน) เหมือนใบเสร็จ 1 ใบมีสินค้าได้หลายบรรทัด

| คอลัมน์ | Prisma | SQLite | NULL? | default | ความหมาย |
|---|---|---|---|---|---|
| `id` | `Int` (PK) | INTEGER AUTOINCREMENT | ไม่ | auto | เลขรันนิ่งของใบ |
| `doc_no` | `String` (unique) | TEXT | ไม่ | — | เลขที่ใบที่มนุษย์อ่านได้ เช่น `"ISS-6907-0001"` — **แอปเป็นคนออกเลขเอง** database บังคับแค่ห้ามซ้ำ |

**กติกาที่แอปต้องคุมเอง (database บังคับให้ไม่ได้):**

## 6.1 Flow ขอเบิก–อนุมัติ (ISSUE) + ตาราง `stock_request_items`

**ที่มา (การตัดสินใจข้อ 9 — คุยกับผู้พัฒนา ก.ค. 2026):** ฝั่งเบิกออก ไม่ว่าจะกดขอล่วงหน้าผ่านเว็บ หรือเดินมาขอที่ห้องสด **ต้อง "ขอผ่านแอป" เหมือนกันหมด** แล้วคนคลัง (role=STAFF) เป็นคนกดยืนยันให้ พร้อมเก็บชื่อผู้ขอติดไปด้วย — ฝั่งรับเข้า (RECEIVE) คนคลังจัดการเองทั้งหมดเหมือนเดิม ไม่ผ่าน flow นี้

> **หมายเหตุฉบับแก้ (8 ก.ค. 2026 — repo web เป็นเจ้าของกติกาแล้ว ตัวจริงอยู่ `DATABASE.md` ข้อ 6):**
> `role` ใช้ 4 ค่า `Admin | Manager | Operator | Viewer` (Admin/Manager = สิทธิ์ระดับ "คนคลัง/STAFF"
> ของเอกสารนี้, Operator = ผู้ขอเบิก/REQUESTER, Viewer = ดูอย่างเดียว — ฉบับแก้ 10 ก.ค. 2026
> `DATABASE.md` ข้อ 6.13) · ผู้ขอกดยกเลิกใบของตัวเองได้เฉพาะตอนยัง PENDING ·
> ป้าย "ปฏิเสธ/ยกเลิก" ไม่เก็บลง database — คำนวณจาก `resolved_by` เทียบ `requested_by` ตอนแสดงผล

**ตาราง `stock_request_items` — รายการที่ขอ (ลูกของใบ ISSUE ที่ยัง PENDING)** ว่างเปล่า ณ ส่งมอบ (0 แถว):

| คอลัมน์ | Prisma | SQLite | NULL? | default | ความหมาย |
|---|---|---|---|---|---|
| `id` | `Int` (PK) | INTEGER AUTOINCREMENT | ไม่ | auto | เลขรันนิ่งของบรรทัดที่ขอ |
| `document_id` | `Int` (FK) | INTEGER | ไม่ | — | ชี้ไป `stock_documents.id` ของใบ ISSUE ที่ขอ |
| `item_id` | `String` (FK) | TEXT | ไม่ | — | ชี้ไป `items.item_id` — ขอสินค้าตัวไหน |
| `qty_requested` | `Float` | REAL | ไม่ | — | จำนวนที่ผู้เบิกกรอกตอนกดขอ — **เก็บถาวร ห้ามแก้ทับ** (อ้างอิงคำขอเดิมย้อนหลังได้) |
| `qty_confirmed` | `Float?` | REAL | ได้ | NULL | จำนวนที่คนคลังให้จริงตอนยืนยัน — NULL ตราบใดที่ยัง PENDING/CANCELLED |
| `note` | `String?` | TEXT | ได้ | NULL | หมายเหตุระดับบรรทัด |

**ทำไมต้องมีตารางนี้แยกจาก `stock_transactions`:** เพราะยอดคงเหลือ = `SUM(qty_change)` **ทันทีที่มีแถว** (กฎเหล็กข้อ 8) ถ้าสร้างแถวใน `stock_transactions` ตั้งแต่ตอนกดขอ ยอดจะลดทันทีทั้งที่ของยังไม่ออกจากชั้นจริง จึงต้องพักคำขอไว้ที่นี่ก่อน — `stock_transactions` เก็บเฉพาะ "การเคลื่อนไหวที่เกิดจริงแล้ว" เท่านั้น คำขอที่ยัง PENDING จะไม่ปนในยอด

**ทำไมแยก `qty_requested` กับ `qty_confirmed`:** คนคลังให้ของไม่ครบตามที่ขอได้ (ขอ 10 มีของ 7) ถ้าแก้ทับช่องเดียวจะไม่เหลือหลักฐานว่าตอนแรกขอเท่าไหร่ — เก็บสองช่องคือ conservative ที่สุด (ตรงหลัก "ห้ามทำข้อมูลเดิมหาย")

**State ของใบ `stock_documents.status` — ครบทุกกรณี:**

| status | doc_type | เกิดจาก | `stock_transactions` | `requested_by` | `resolved_by`/`resolved_at` |
|---|---|---|---|---|---|
| `CONFIRMED` | RECEIVE | คนคลังบันทึกรับเข้าตรง ๆ | สร้างทันทีตอนสร้างใบ | NULL เสมอ | NULL ทั้งคู่ |
| `PENDING` | ISSUE | มีคนขอเบิก (เว็บ/หน้าห้อง) | **ยังไม่มี** — พักที่ `stock_request_items` | ผู้เบิก (บังคับมี) | NULL ทั้งคู่ |
| `CONFIRMED` | ISSUE | คนคลังกดยืนยัน + กรอก `qty_confirmed` | สร้าง ณ ตอนนี้ (`type=OUT`, `qty_change = -qty_confirmed`, `created_by`=คนคลัง) | ผู้เบิก | คนคลัง + เวลานั้น |
| `CANCELLED` | ISSUE | คนคลังกดยกเลิกคำขอ | **ไม่มีเลยตลอดกาล** | ผู้เบิก | คนคลัง + เวลานั้น |

`PENDING`→`CONFIRMED` และ `PENDING`→`CANCELLED` เป็น terminal ทั้งคู่ (ปิดแล้วปิดเลย ห้ามย้อน)

**ชื่อผู้เบิกติดกับรายการยังไง:** ไม่ต้องเพิ่มคอลัมน์ที่ `stock_transactions` — join ย้อน `stock_transactions.document_id → stock_documents.requested_by → users.name` ได้เลย

**`transaction_date` ของรายการที่ยืนยัน (ต่างจาก RECEIVE):** ของออกจากคลังจริง*ตอนคนคลังกดยืนยัน/ส่งมอบ* ไม่ใช่ตอนกดขอ (เคสจองล่วงหน้าอาจห่างกันหลายวัน) — จึงควรลง `transaction_date` = วันที่ยืนยัน (เช่น วันเดียวกับ `resolved_at`) **ไม่ใช่ `doc_date` ซึ่งเป็นวันที่ขอ** ต่างจากใบ RECEIVE/ใบตรงที่ `transaction_date = doc_date` (สร้างรายการพร้อมใบ) — เป็น business rule ที่แอปกำหนด (ยังปรับได้ที่ repo web ถ้ามีเหตุผลอื่น)

**ข้อจำกัดที่รู้ตัว — ยังไม่มีระบบ "กันยอดจอง":** คำขอ PENDING ไม่สร้าง transaction จึงล่องหนจากยอดคงเหลือ*และจากคำขอใบอื่น* → 2 คนจองของชิ้นสุดท้ายพร้อมกันได้ ระบบไม่เตือน ชนกันตอนคนคลังกดยืนยันคนที่ 2 (เห็นชั้นว่างแล้วปรับ `qty_confirmed`/ยกเลิก) — **เวอร์ชันแรกตั้งใจให้คนคลังเป็นด่านจับตรงนี้** ถ้าอนาคตอยากได้ "ยอดพร้อมจ่าย = ยอดคงเหลือ − ยอดที่ถูกจองค้าง" ค่อยเพิ่ม logic ทีหลัง (ไม่ต้องแก้ schema)

## 8. กฎเหล็ก: ยอดคงเหลือเป็นค่าคำนวณ

**ยอดคงเหลือของสินค้า = ผลรวม `qty_change` ทุกแถวของสินค้านั้น — ไม่มีคอลัมน์เก็บยอดที่ไหนทั้งสิ้น**

SQL ที่ถูกต้อง (จำท่านี้):

```sql
SELECT COALESCE(SUM(qty_change), 0) AS balance
FROM stock_transactions
WHERE item_id = ?;
```

ผ่าน Prisma:

```js
const r = await prisma.stockTransaction.aggregate({
  _sum: { qty_change: true },
  where: { item_id: itemId },
});
const balance = r._sum.qty_change ?? 0; // aggregate ให้ null ถ้าไม่มีแถว — ต้อง ?? 0 เสมอ
```

**ทำไมต้อง `COALESCE`/`?? 0`:** สินค้า 2,382 ตัว แต่มี transaction แค่ 579 ตัว — **อีก 1,803 ตัวไม่มีแถวเลย** (ตอน migrate ตั้งใจไม่สร้างแถวยอดศูนย์) และ `SUM()` ที่ไม่เจอแถวให้ค่า `NULL` ไม่ใช่ `0` ถ้าลืมกันตรงนี้ สินค้าเกือบทั้งคลังจะแสดงยอดเป็น null/undefined แทนที่จะเป็น 0

ยอดรวมทั้งคลัง ณ วันส่งมอบ = **3,016.85** (ใช้เช็คความถูกต้องได้ — ดู [`etl/out/verify_report.md`](../etl/out/verify_report.md) เช็คข้อ 2)
